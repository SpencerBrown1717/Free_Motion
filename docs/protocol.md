# Command protocol

Free Motion's wire protocol between **OpenClaw** (or any client) and a **device**.

## Status

**v0 — locked.** No breaking changes without bumping `v`. Field additions that don't break existing parsers are allowed and don't require a version bump.

## Goals

- Stable contract that survives swapping transports (Telegram now, MQTT/HTTP/serial later).
- Every command carries an explicit **safety mode**.
- Easy for a human to send by hand, easy for code to send.
- A single internal `Command` representation on the device, no matter what the wire form was.

## v0 decisions

These were resolved before M1 code landed. Each one is binary; the rationale is short.

- **Telemetry channel?** **In-band.** Devices emit unsolicited frames using the [reply envelope](#reply-envelope) shape with a fresh `id`. Clients ignore unknown ids. A separate channel only happens when telemetry rate makes Telegram impractical.
- **`to` field?** **Optional.** Telegram is 1:1 today. If absent, the device assumes the message is for it. Becomes required when fan-out lands.
- **Args schemas?** **Inline in this file** for v0. Args today are tiny. Move to per-command JSON Schema files under `docs/protocol/` when any command's args exceed a few fields.
- **Wire-level dedup?** **Trust transport order, dedupe by `id`.** Devices SHOULD ignore commands whose `id` they have processed in the last 60 seconds. No sequence numbers.
- **Slash sugar AND JSON?** **Both.** Same internal `Command`. Slash sugar is a strict subset of JSON.
- **Correlation id == Telegram message id?** **No.** `id` is a UUID generated by the sender. Mapping to a transport-specific id would couple us to that transport.

## Transport (v0)

Telegram text messages. Two equivalent forms travel on the same channel:

- **Slash sugar** — human-friendly: `/ping`, `/led on`, `/status`. The device translates these to envelopes internally.
- **JSON envelope** — programmatic: a single Telegram text message containing one JSON object. OpenClaw and other clients use this form.

The device parses both into the same internal `Command` value.

## Command envelope

```json
{
  "v": 0,
  "id": "8c3b1a3e-7a90-4b2a-9c19-1f1f5b0a8b21",
  "ts": "2026-05-03T16:30:00Z",
  "from": "openclaw",
  "to": "pi-bench-01",
  "cmd": "ping",
  "args": {},
  "safety": "dry_run"
}
```

| Field | Type | Required | Notes |
|---|---|---|---|
| `v` | int | yes | Protocol version. Always `0` for v0. |
| `id` | string (UUID) | yes | Correlation id. The device echoes it on the reply. |
| `ts` | string (RFC 3339, UTC) | yes | Sender clock. Devices may reject stale timestamps in `live` mode. |
| `from` | string | yes | Free-form sender id (e.g. `openclaw`, `chat:12345`). |
| `to` | string | **no** | Device id, or `*` for broadcast. Absent means "this device". |
| `cmd` | string | yes | Enum, see [Allowed commands](#allowed-commands-v0). |
| `args` | object | yes | Command-specific. Empty object if none. |
| `safety` | enum | yes | `dry_run` \| `bench` \| `live`. See [Safety modes](#safety-modes). |

## Reply envelope

```json
{
  "v": 0,
  "id": "8c3b1a3e-7a90-4b2a-9c19-1f1f5b0a8b21",
  "ts": "2026-05-03T16:30:00Z",
  "from": "pi-bench-01",
  "ok": true,
  "error": null,
  "state": "idle",
  "telemetry": {},
  "message": "pong"
}
```

| Field | Type | Required | Notes |
|---|---|---|---|
| `v` | int | yes | Same as the command. |
| `id` | string | yes | Echoes the command's `id`, **or** a fresh UUID for unsolicited frames (see below). |
| `ts` | string | yes | Device clock. |
| `from` | string | yes | Replying device id. |
| `ok` | bool | yes | Whether the command was accepted and executed. |
| `error` | object \| null | yes | `null` when `ok` is `true`; otherwise `{ "code": "...", "message": "..." }`. |
| `state` | string | yes | Device state after the command (e.g. `idle`, `armed`, `moving`, `error`). |
| `telemetry` | object | yes | Optional structured fields. Empty object if none. |
| `message` | string | yes | Human-readable summary. Safe to show in a chat. |

### Unsolicited telemetry

Devices may emit reply-shaped frames without a matching command (startup banners, periodic heartbeats, async events). These frames use the same envelope, with `id` set to a fresh UUID and `ok=true`. Clients that don't recognize an `id` should treat the frame as informational and not error on it.

## Allowed commands (v0)

Intentionally tiny. Every additional command needs a tracked rationale.

| `cmd` | `args` | Reply `state` | Honored in `dry_run` | Notes |
|---|---|---|---|---|
| `ping` | `{}` | `idle` | yes | Round-trip check. `/ping` sugar. |
| `status` | `{}` | unchanged | yes | Host info. `/status` sugar. |
| `capabilities` | `{}` | unchanged | yes | Returns the device's capability set. |
| `led_on` | `{ "pin"?: int }` | unchanged | logged, not actuated | Optional pin override. |
| `led_off` | `{ "pin"?: int }` | unchanged | logged, not actuated | |
| `arm` | `{}` | `armed` | refused (`unsafe_in_mode`) | Requires `bench` or `live`. |
| `disarm` | `{}` | `idle` | yes | Always allowed. |
| `stop` | `{}` | `idle` | yes | **Hard stop, never refused, in any mode.** |
| `move` | `{ "x": float, "y": float, "z": float }` | `moving` | logged, not actuated | Relative offset from current position. Slash sugar: `/move x y z`. |

## Safety modes

- **`dry_run`** — Device parses, validates, and logs every command, but does not actuate hardware. Default when ambiguous on the protocol level.
- **`bench`** — Low-energy actions allowed (LEDs, low-RPM motors with no load, servos out of vehicle). Intended for desk testing.
- **`live`** — Real actuation in the real world. Device may refuse `live` if its config does not opt in.

When a sender omits `safety` (only possible in slash sugar), the device fills it in from its **config default** (`FREEMOTION_SAFETY_DEFAULT`, defaults to `dry_run`).

Hard rules:

1. `stop` is honored in any mode, with no auth or argument requirements beyond a valid envelope.
2. A device may refuse `arm` if it judges itself unsafe (low battery, missing config, watchdog tripped). Refusal returns `ok=false`, `error.code="unsafe_in_mode"` or a more specific code.
3. Anything that **could** move hardware must check both auth and `safety` before acting.

## Device registration

On startup the agent emits one self-describing reply-shaped frame to the configured upstream chat:

```json
{
  "v": 0,
  "id": "<uuid>",
  "ts": "...",
  "from": "<device id>",
  "ok": true,
  "error": null,
  "state": "idle",
  "telemetry": {
    "device_id": "pi-bench-01",
    "hardware": "pi",
    "software_version": "0.1.0",
    "capabilities": ["ping", "status", "capabilities", "led_on", "led_off", "stop"],
    "safety_default": "dry_run"
  },
  "message": "device online"
}
```

`capabilities` lists the subset of [Allowed commands](#allowed-commands-v0) the device actually implements. Clients should treat anything outside this list as unsupported on that device.

## Auth (v0)

- **Telegram chat allowlist** — `TELEGRAM_ALLOWED_CHAT_IDS` (already in M0).
- **Per-command allow/deny** — planned for M2 in the agent's config.
- Non-Telegram transports may add HMAC signing later; out of scope for v0.

## Errors

Stable error codes (extend deliberately):

| `error.code` | Meaning |
|---|---|
| `unauthorized` | Sender not allowed by the device's auth rules. |
| `unknown_cmd` | `cmd` not recognized. |
| `bad_args` | Envelope parsed but `args` (or another field) failed validation. |
| `unsafe_in_mode` | Command refused because of the current `safety` mode. |
| `device_busy` | Device cannot accept the command in its current state. |
| `internal` | Unhandled error inside the device. |

Replies always include `error.message` (human-readable) alongside `error.code` (stable).

## Examples

### Slash sugar → envelope

A user typing `/ping` in Telegram is equivalent to:

```json
{
  "v": 0, "id": "<uuid>", "ts": "<now>",
  "from": "chat:12345",
  "cmd": "ping", "args": {}, "safety": "<device default>"
}
```

`/led on` (the device's `safety_default` is whatever it was started with):

```json
{
  "v": 0, "id": "<uuid>", "ts": "<now>",
  "from": "chat:12345",
  "cmd": "led_on", "args": {}, "safety": "<device default>"
}
```

### Hard stop

A `stop` reply, regardless of prior state:

```json
{
  "v": 0, "id": "<uuid>", "ts": "<now>",
  "from": "pi-bench-01",
  "ok": true, "error": null,
  "state": "idle", "telemetry": {},
  "message": "stopped"
}
```
