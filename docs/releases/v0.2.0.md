# Free Motion v0.2.0 ‚Äî Pi-first lockdown

**Tag.** [`v0.2.0`](https://github.com/SpencerBrown1717/Free_Motion/releases/tag/v0.2.0)
**Date.** 2026-05-03
**Predecessor.** [`v0.1.0-alpha`](https://github.com/SpencerBrown1717/Free_Motion/releases/tag/v0.1.0-alpha) (the first runnable cut, mocks-only).

This release is the Pi-first lockdown: every adapter on the documented Pi reference path is real (not mocked), the contract that those adapters honor is locked in [`docs/pi-reference.md`](../pi-reference.md), and a single named operator-runnable benchmark ‚Äî `pi_follow_bench` ‚Äî verifies that the contract holds end-to-end on a real Pi. The release is the gate for [M5 Phase 1 (Jetson Nano)](../../ROADMAP.md#what-to-build-next).

If you used `v0.1.0-alpha`, your code still runs unchanged: the protocol is `v: 0`, the eight commands `/ping /capabilities /status /arm /disarm /move /mission_start /stop` are exactly what the locked surface registers, and the deny list, mock backends, and laptop demos all still work the same way. Everything new is additive.

---

## What is locked

This is the part of the project a contributor or downstream user can now depend on without expecting it to move under them. Every item below has a documented contract (in `docs/`), a code home (in `freemotion/` or `examples/`), and a test suite covering the contract.

- **Protocol v0** ‚Äî typed `Command` and `Reply` envelopes, slash sugar plus JSON, `v: 0`. Same shape as `v0.1.0-alpha`. ([`docs/protocol.md`](../protocol.md))
- **Eight-command surface** ‚Äî `/ping`, `/capabilities`, `/status`, `/arm`, `/disarm`, `/move`, `/mission_start`, `/stop`. Anything outside this set on the Pi reference path is out of scope. ([`docs/pi-reference.md` ¬ß2](../pi-reference.md))
- **Pi hardware path** ‚Äî `PiHardwareController` on Pi 4/5 (Pi 3 with caveats), Bookworm or newer, BCM mode, `armed_pin = 27`, `moving_pin = 22`, Pi camera via `picamera2` / libcamera at `(640, 480)`. ([`docs/pi-hardware.md`](../pi-hardware.md), [`docs/pi-reference.md` ¬ß3](../pi-reference.md))
- **Model path** ‚Äî `PiCameraSource ‚Üí YoloVision ‚Üí WorldState ‚Üí GemmaMissionControl ‚Üí SafetyGate ‚Üí PiHardwareController`. Defaults: `yolov8n.pt`, `gemma-2-2b-it`. ([`docs/models.md`](../models.md), [`docs/pi-reference.md` ¬ß4](../pi-reference.md))
- **Env-var contract** ‚Äî five tiers from required (`TELEGRAM_BOT_TOKEN`) through demo-only knobs. ([`docs/pi-reference.md` ¬ß5](../pi-reference.md))
- **Safety contract ‚Äî twelve guarantees** ‚Äî `dry_run` cannot actuate, `bench` allows only the bench-safe primitive, `/stop` is unconditional, stale-world refusal, degraded summary, hung-tick handling, ordered graceful shutdown. ([`SAFETY.md`](../../SAFETY.md), [`docs/pi-reference.md` ¬ß6](../pi-reference.md), [`docs/pi-failure-modes.md`](../pi-failure-modes.md))
- **Status contract** ‚Äî `/status` exposes `controller` + `mission_loop` telemetry; the human-readable mission line format is `mission: <state> [DEGRADED: ...] [stale world: ...] [(intent='...')]`. ([`docs/pi-reference.md` ¬ß7](../pi-reference.md))
- **Failure model** ‚Äî every documented environmental failure (camera, vision, mission, dispatch, signals, network) has a behavior + a `/status` signal + an operator action. ([`docs/pi-failure-modes.md`](../pi-failure-modes.md))
- **Benchmark protocol ‚Äî `pi_follow_bench`** ‚Äî frozen 10-step sequence, 8 first-class pass/fail flags, JSON artifact schema v1. The single operator-runnable proof that the locked contract still holds. ([`docs/pi-benchmark.md`](../pi-benchmark.md), [`examples/pi_follow_bench/README.md`](../../examples/pi_follow_bench/README.md))
- **M5 acceptance gate** ‚Äî a Jetson port is "done" only when a Jetson rig produces a `pi_follow_bench`-shaped artifact (rename to `jetson_follow_bench` is allowed; the schema, sequence, and criteria are not). ([`docs/pi-reference.md` ¬ß10](../pi-reference.md))

Architectural decision records: ADR-0001 through ADR-0013, all in [`docs/decisions.md`](../decisions.md).

## What is still open

Honest list ‚Äî these are explicitly **not** locked by `v0.2.0` and may move in subsequent releases.

- **Jetson Nano support (M5 Phase 1)** ‚Äî the Pi contract is locked specifically to make this port deterministic. Phase 1 plans (the must-keep / allowed-to-differ / first-target-demo / acceptance-gate doc) and the Jetson dependency mapping are next.
- **ESP32 / Arduino (M5 Phase 2 / 3)** ‚Äî deferred behind Jetson; will be a separate ADR set.
- **Operator authentication** ‚Äî agent treats every Telegram message from a verified bot session as authoritative. No allow lists, rate limits, watchdogs, or link-loss fail-safes ship in `v0.2.0`. Out of scope for the Pi-first lockdown; tracked as future safety work.
- **Multi-device fan-out** ‚Äî the protocol's optional `to` field is reserved but unused; `from` is sender-only. One agent talks to one device.
- **Second transport** ‚Äî Telegram is the only transport. The protocol is transport-agnostic; another transport (HTTP, MQTT, gRPC) hasn't been written.
- **Hardware autostart on boot** ‚Äî systemd unit files ship for `pi_bench_demo`, `pi_closed_loop_demo`, and `pi_follow_bench`, but the operator is responsible for installing and enabling them per [`examples/pi_closed_loop_demo/README.md`](../../examples/pi_closed_loop_demo/README.md). Auto-recovery of camera handles after USB unplug is not implemented (deliberate ‚Äî see [ADR-0011](../decisions.md)).
- **CI hardware runner** ‚Äî CI runs `pi_follow_bench --mode=ci` on every push; bench-mode runs (real Pi) are the operator's responsibility. There is no hosted Pi runner.

## What's new since `v0.1.0-alpha`

In rough shipping order. Each line below has a CHANGELOG section with full detail and a corresponding ADR.

1. **M4 Phase 1 ‚Äî `PiHardwareController` + factory + config hook.** First real hardware adapter. Lazy `RPi.GPIO` import; `armed_pin` / `moving_pin` defaults; offline fallback when GPIO setup fails. ([`docs/pi-hardware.md`](../pi-hardware.md), ADR-0004)
2. **M4 Phase 2 ‚Äî `pi_bench_demo`.** First end-to-end Telegram ‚Üí Pi GPIO path. ([`examples/pi_bench_demo/`](../../examples/pi_bench_demo/))
3. **M4 Phase 3 ‚Äî `SafetyGate`.** Composition wrapper that enforces the device's `safety_default` as the floor: per-command overrides cannot escalate above the device's mode. `dry_run` cannot actuate; `/stop` always passes through. ([`docs/decisions.md` ADR-0006](../decisions.md))
4. **M4 Phase 4 ‚Äî docs polish.** `docs/pi-hardware.md`, `GETTING_STARTED.md`, README, ROADMAP, CHANGELOG aligned around the M4 reality.
5. **post-M4 ‚Äî `YoloVision`.** First real perception adapter. Lazy `ultralytics` import; person-only by default; bbox locked to corner-based normalized `(x, y, w, h)`; caller-injected `frame_source`. ([`docs/models.md`](../models.md), ADR-0007)
6. **post-M4 ‚Äî `GemmaMissionControl`.** First real decision adapter. Lazy `transformers` import; tolerant JSON parser; one decision per `plan()` call; offline fallback. ([`docs/models.md`](../models.md), ADR-0008)
7. **Step 1 ‚Äî `PiCameraSource`.** Canonical Pi camera frame producer for the existing `YoloVision(frame_source=...)` seam. Lazy `picamera2` import; transient-failure tolerant; idempotent `close()`; lock-free per-call. ([`docs/pi-camera.md`](../pi-camera.md), ADR-0009)
8. **Step 2 ‚Äî `MissionLoop` + `pi_closed_loop_demo`.** Background daemon thread runs `vision ‚Üí world ‚Üí mission ‚Üí router.dispatch` per tick. Only `MOVE` is dispatched from the loop. Per-stage failure isolation. `/stop` interrupts mid-tick. ([`docs/pi-closed-loop.md`](../pi-closed-loop.md), ADR-0010)
9. **Step 3 ‚Äî Failure-mode hardening.** Stale-world refusal (`stale_world_timeout_s`), per-stage consecutive failure counters with a single `degraded` flag, hung-tick handling, ordered graceful shutdown (`mission_loop.stop` first, then controller, then camera). ([`docs/pi-failure-modes.md`](../pi-failure-modes.md), ADR-0011)
10. **Step 4 ‚Äî Pi reference architecture lock.** Every surface from ¬ß1‚Äì10 above named, frozen, and locked in [`docs/pi-reference.md`](../pi-reference.md). M5 = "same contract, different hardware." Zero new code. ([ADR-0012](../decisions.md))
11. **Step 5 ‚Äî `pi_follow_bench`.** The named, repeatable Pi benchmark demo. Direct router dispatch (no Telegram for reproducibility), frozen 10-step sequence, frozen pass/fail criteria, JSON artifact schema v1, three failure injects (`camera_offline`, `mission_offline`, `vision_drop_after_n`), CI-mode for per-push verification. The M5 acceptance gate. ([`docs/pi-benchmark.md`](../pi-benchmark.md), [`examples/pi_follow_bench/README.md`](../../examples/pi_follow_bench/README.md), ADR-0013)

## Upgrade notes

- **Protocol is unchanged** ‚Äî `v: 0`. Existing JSON envelopes round-trip identically.
- **No breaking changes to public APIs.** Every addition behind a config flag (`FREEMOTION_VISION_BACKEND=yolo|mock`, `FREEMOTION_MISSION_BACKEND=gemma|mock`, `FREEMOTION_HARDWARE=pi|mock`). Default everywhere is `mock`, so existing deployments built against `v0.1.0-alpha` keep working with no changes.
- **New optional extras:** `pip install -e .[yolo]`, `[gemma]`, `[picam]`. Base install stays stdlib + `python-telegram-bot`.
- **New env vars (all optional):** `FREEMOTION_PI_ARMED_PIN`, `FREEMOTION_PI_MOVING_PIN`, `FREEMOTION_VISION_BACKEND`, `FREEMOTION_MISSION_BACKEND`, `FREEMOTION_MISSION_TICK_INTERVAL_S`, `FREEMOTION_DEFAULT_INTENT`. Documented in [`docs/pi-reference.md` ¬ß5](../pi-reference.md).
